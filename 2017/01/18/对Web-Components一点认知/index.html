对Web Components一点认知

Jan 18, 2017 | 组件化 | Hits

Template

使用层面

  1. 获取模板
  2. 添加模板到指定节点

认知层面

这个有点类似script标签来定义可复用的模板,只不过现在有个专门的标签来表示模板了。不过在我看来是一个东西。但是看到template不仅仅可以定义html。
尽然还可以内嵌样式。

有点疑惑:

  1. 内嵌样式是局部作用的吗?
    • 全局污染的
  2. script方式可以定义样式吗,局部作用域(不污染全局)的吗?
    • 可以定义样式,type需要自定义,否则会报错
    • 全局污染的

小结: 起码到现在为止通过实例更加肯定了这货没什么特别的,说到底是跟script来定义模板。是一个东西。如果硬要说有差别:

  1. 使用接口上略有不同
  2. 定义方式上略有不同

Custom Element(自定义元素)

很显然从直观上理解这东东,很容易让人与现在使用MVV*框架定义组件来作比较。

MVV*组件化

先来谈谈相对比较熟悉的MVV*组件化的优势:

  1. 页面的可读性增强
  2. 代码的复用性增强

相反,劣势:

很简单就是不跨框架。不够通用。因为组件化的方式并非是标准。

谈谈MVV*组件话需要处理的几个核心方面:

  1. 组件之间的通信
  2. 组件的行为和事件
  3. 组件之间的嵌套
  4. 组件的生命周期

从MVV*组成的角度来看看有哪些部分组成:

  1. javascript
  2. html
  3. css

Custom Element

认知层面:

似乎Custom Element是基于原生HTMLElement的,扩展了对应的接口。

那我们先来分析原生HTMLElement,看看有哪些接口:

  1. 行为和事件 -> action 和 onclick、mouseover…
  2. 具备生命周期的回调函数.

以上也是Custom Element可扩展的地方。

看到这里,我感觉自定义元素的出发点无非是使得原生元素变得可继承。现在看它能为组件化出多大力,我是没看出半点来。

小结: Custom Element可能在使用上与MVV*定义的组件有极为相似的一点(注意措辞,一点)。但是说到底两个的出发点是截然不同的。无论是组成还是使用场景。。

Shadow Dom

影子dom? 好吧从名称上是看不出任何其想要解决组件化的决心。

看到这里才明白,无论是Shadow Dom还是Custom Element还是Template都是为了实现标准组件化的基础。与MVV*组件化对比得深刻理解了这些基础,再作总结对比。

Shadow DOM解决了将一个dom节点变成由css + js + html组成的节点。严格来讲是(Template + Shadow Dom)的组合达到了这样的效果,且样式不是全局污染的。现在很多MVV*组件化中css虽然是物理、逻辑分离的,但是仍旧是全局
污染的。从这角度看,这样的组件化是存在一定瑕疵的。

小结: Shadow DOM说到底也是dom的一种类型节点,无非这种节点更高级,可以包含js、html、css。这位组件的标准化奠定了一定基础。

Import HTML

不深究,肤浅的认识两点:

  1. 导入HTML可以包含js\html\css的「组件」
  2. 可以不需要js来加载html了,只需要定义<link rel="import" href="dialog.html">这些形式便可。

Web Component

Template + Custom Element + Shadow Dom + Import HTML = Web Component ?

从Web Components最后一个实例来看,可以看出这几种技术的组合产生了神奇的效果。标准的组件化并非不切实际。只不过由于本文缺少典型的实例,缺少了对其进一步认知的机会,当然也由于篇幅有限,对Web Components的理解先止于此。

小结

  • Template : 定义模板,模板可包含css、js、html,当然如果组件化,物理的划分通常不会以template而是以html文件。
  • Custom Element : 单独搞,确实没什么意义,但如果与Shadow DOM和Template结合,就很酷了,说通俗点可以自定一个dom节点包含了css\js\html而且还是局部的,无侵入的。
  • Shadow DOM : 牛逼,可以让一个节点抽象成挂载了js/html/css。啥也不说了。
  • Import HTML : 组件化的粒度通常是比较大的,这里其实可以看出最小的组件粒度就是html文件。而且使用了import方式,非常方便、灵活。

BTW: 接下来就是实践了,希望能够加深对Web Components的理解。
